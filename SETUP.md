# Flow 繪製工具 — 雲端版部署步驟（Vercel + Supabase）

這份說明是給 Jerry（或代為設定的人）看的，跟基層/主管實際使用的操作方式無關 —— 他們只要打開網址就能用，不需要看這份文件。

## 你會用到的檔案

- `index.html` — **現在唯一要發給大家的網址**。進去就是一個統一登入頁（選單位＋輸入PIN），依PIN決定是進到員工畫圖模式、單一單位主管彙整模式、還是Master跨單位總覽模式（見下方「登入身份」一節）。
- `flow-builder.html` / `manager-view.html` — 舊版的兩個獨立檔案（員工版／主管版分開），已被 `index.html`取代，留著只是備份，不需要再發這兩個網址給任何人。
- `supabase-schema.sql` — 建立資料庫用的 SQL；`master-support.sql` — 新增 Master 跨單位總覽功能用的 SQL（第一次設定時兩份都要在 Supabase SQL Editor 跑過）。

## 步驟 1：建立 Supabase 專案

1. 到 [supabase.com](https://supabase.com) 用你的帳號建立一個新專案（免費方案就夠用，~100人的資料量很小）。
2. 進專案後，左側選單找 **SQL Editor**，新增一個 query，把 `supabase-schema.sql` 整份內容貼進去，按 Run。
   - 如果你的單位名稱不是「DIV4」，把 SQL檔最後那行 `insert into unit_pins ...` 的單位名稱和PIN改成你實際要用的，或直接在 Supabase 的 **Table Editor** 裡對 `unit_pins` 這張表手動新增每個單位一列（單位名稱、PIN）。
   - 有幾個單位就新增幾列；之後要換PIN，直接在Table Editor改那一格就好，不用重新跑SQL。
   - **單位名稱務必跟填寫者存檔時填的單位名稱完全一致**（工具會自動把英文轉大寫、去掉頭尾空白，所以「Div4」「div4」「 DIV4 」都會變成「DIV4」——`unit_pins`裡的名稱也要用「DIV4」這個大寫形式，才能對上）。
3. 左側選單找 **Settings → API**，記下兩個值：
   - **Project URL**（例如 `https://xxxxx.supabase.co`）
   - **anon public** 這組 key（一長串字串，不是 service_role key，那組絕對不要放進前端檔案）

## 步驟 2：把連線資訊填進 index.html

打開 `index.html`，搜尋 `SUPABASE_URL` 那兩行（在 `<script>` 區塊很前面的地方），把空字串換成步驟1記下的值：

```js
const SUPABASE_URL = 'https://xxxxx.supabase.co';
const SUPABASE_ANON_KEY = '你的anon public key';
```

anon key是設計成可以放在前端程式碼裡公開的（真正的存取控制是靠 SQL 裡設定的 RLS 政策和 PIN 檢查函式，不是靠隱藏這個 key），不用擔心洩漏。（`flow-builder.html`/`manager-view.html` 這兩個舊檔案裡也有同樣的值，已經一起改好，但既然不再對外發連結，只是保留備份而已。）

## 步驟 3：部署到 Vercel

最簡單的方式（不需要 git/程式背景）：

1. 到 [vercel.com](https://vercel.com) 登入。
2. 新建一個 Project，選擇「Deploy without Git」或直接把整個資料夾拖進去（Vercel 支援直接拖拉一個資料夾部署純靜態網站）。
3. 把整個「Flow 繪製工具」資料夾（含 `index.html`）一起上傳——多帶著 `.sql`/`.md` 檔沒關係，只有 `.html` 會被當網頁提供。
4. 部署完成後，Vercel 會給你一個網址，例如 `https://your-project.vercel.app`——因為主入口叫 `index.html`，這個網址本身（不用再接檔名）就會直接打開登入頁。
5. 把這一個網址發給所有人（員工／主管／Master 都用同一個網址，進去後靠PIN分流），不用再像以前一樣管理兩個不同連結。

## 帽子（六階段schema）設定 —— 各單位主管自行定義

不同BU的工作流程不一樣（例如你的Div4可能6個帽子就夠，業務BU、行銷BU可能需要完全不同的階段），所以帽子**綁在單位上**，不是全公司共用一套：

- 用該單位的PIN登入 `index.html`（不是 asus）進到彙整畫面後，點右上角「⚙ 帽子設定」，就能新增/刪除/改名/排序/改顏色自己單位的帽子。**Master 模式沒有這個按鈕**——因為Master是跨單位總覽，沒有單一單位的PIN可以拿來寫回，各單位的帽子仍要各單位自己用自己的PIN去設定。
- 不需要另外記一組管理員密碼 —— 就是同一組單位PIN，跟看彙整結果用的是同一個門檻。
- 員工模式（PIN=asus）完全沒有編輯帽子的按鈕或入口，只會讀取自己單位目前設定的帽子來畫圖，讀不到就自動退回內建預設的6個帽子。
- 還沒有任何單位主管設定過帽子時，所有人看到的都是內建預設的6個Request Collection...Go Live，跟舊版行為一致。
- 改標題/顏色不會影響既有資料；**刪除**整個帽子後，已經歸在那個帽子下的舊資料不會消失（畫面上會標成「已移除的帽子」），只是沒辦法再新增內容進去，也選不到了。

## 登入身份 —— 單位下拉 ＋ PIN 決定你進到哪個畫面

`index.html` 進去就是一個登入頁：選單位（Div1~4 或 Master）、輸入PIN、（員工模式還要填姓名／部門角色）。**PIN決定你的角色**，不是分開的網址：

| PIN | 對應角色 | 說明 |
|---|---|---|
| `asus` | 員工（畫圖） | 選 Div1~4 其中一個＋填姓名/部門角色，進去畫自己的flow；右上角「儲存資料」存到雲端。所有單位共用同一組 `asus`，因為這只是進場門檻，不是機密。 |
| 該單位在 `unit_pins` 表裡的真實PIN（目前四個單位都是 `0000`） | 該單位主管彙整 | 只看得到選定那一個單位的彙整結果＋能設定該單位的帽子。 |
| Master專屬PIN（`master-support.sql`裡設定的，目前是 `8888!`） | 跨單位總覽 | 選「Master」，畫面左側出現Div1~4的選單，一次點一個Div查看——**不是**把四個單位攤平混在一起顯示，切換Div時會連同該Div自己設定的帽子一起換過去（跟該Div主管自己登入看到的完全一樣），只是門檻換成Master PIN，不需要知道每個單位各自的PIN。唯讀，沒有帽子設定按鈕（要改帽子還是要該單位自己的主管用自己的PIN去改）。 |

員工登入後，姓名／部門角色會被記住（下次登入自動填好，方便），但**單位跟PIN每次都要重選/重填**——這是刻意的，因為同一個瀏覽器可能被拿來測試不同角色。

**已知取捨（Jerry自己決定的，如實記錄）**：`asus`/`0000`/`8888!`這幾組PIN都刻意設得簡單、而且同單位/同角色共用，因為Jerry認定這份資料不算機密、主管之間互相看到也無妨。如果之後這份工具的內容變敏感，這幾組PIN都需要換成更難猜、更分散（每個人一組）的形式，目前的設計並不能真正防止「知道密碼的人互相看到彼此不該看的東西」。

## 之後要怎麼維護

- **換單位PIN**：Supabase Table Editor 開 `unit_pins` 表，改那一格即可，不影響已經存的資料或帽子設定。
- **換Master PIN**：同一張 `unit_pins` 表裡 `unit_name='MASTER'` 那一列。
- **新增單位**：在 `unit_pins` 表新增一列（單位名稱要用跟填寫者一致的正規化名稱，通常就是英文全大寫），`stages` 欄位留空即可；也要記得把 `index.html` 登入頁的 `<select id="loginUnitSelect">` 加一個對應的 `<option>`。
- **有人反應存檔失敗**：多半是還沒填 SUPABASE_URL/ANON_KEY，或 Supabase 專案本身被暫停（免費方案閒置一段時間可能會被自動暫停，需要進 Supabase 後台喚醒）。
- **有人反應帽子跑掉了/跟別人不一樣**：先確認他登入時選的「單位」是不是跟主管設定時用的單位不一致。
- **`flow-builder.html`/`manager-view.html` 這兩個舊檔案**：已經不對外發連結，但程式碼還在，純粹當作備份／參考；`index.html` 才是唯一實際維護、實際發給大家用的檔案，之後修bug只改 `index.html` 就好。

## 已知的限制（如實告知，不是要嚇你，是讓你判斷夠不夠用）

- 身份辨識只靠「姓名＋單位＋部門角色」自己填，沒有帳號密碼 —— 同單位同部門角色下如果剛好有兩個人同名，會被當成同一人、後存的覆蓋先存的。目前規模（~100人分散在不同單位/角色）下這個機率不高，但值得知道。
- 主管／Master看到的是「彙整」畫面，不是「即時協作」畫面 —— 員工存檔那一刻的內容才會被看到，中途編輯中的內容看不到，這是你自己選的方向（工程量低），不是bug。
- PIN是輕量門檻，不是完整登入系統：知道PIN的人都能看到對應範圍的彙整結果，PIN本身不記錄「是誰在看」。如果之後有更高的保密需求，才需要再加真正的帳號系統（Supabase Auth）。
- Master模式驗證PIN的方式，是拿你輸入的PIN去嘗試載入「Div1」這個單位——如果Div1剛好還沒有任何人存過資料，畫面會顯示「查無資料」，這時候不代表PIN一定錯，可以點左側其他Div的按鈕再試一次確認（跟一般單位「查無資料」訊息同樣的取捨：故意不區分「PIN錯」跟「PIN對但沒資料」，避免PIN檢查被前端訊息反推出來）。
